# Module Specification Document
## Import & Export Module (Data Exchange)
**System:** KIYAN Factory ERP (Adaptable for Generic Enterprise Manufacturing)
**Prepared by:** Enterprise ERP Solution Architect
**Date:** July 2026
**Architecture Environment:** Modular Monolith, ACID-Compliant, Background-Processing Ready

---

## 1. Objective & Scope
This document outlines the official specification for the Import & Export Module. In an enterprise ecosystem, data rarely lives in a vacuum. The ERP must ingest massive datasets during legacy migrations or daily bulk operations (e.g., importing bank statements, bulk price-list updates) and must extract data securely for external reporting or auditing. This module provides a robust, fail-safe engine for mass data exchange, prioritizing data integrity, strict validation, and airtight security against data exfiltration.

---

## 2. Supported Formats & Modalities

The system supports context-specific file formats tailored to the target audience and data complexity:

*   **Excel (.xlsx):** The primary format for business users. Preserves data types (Dates, Currencies, Booleans), supports multiple sheets, and allows the ERP to generate templates with built-in dropdown validations for the user.
*   **CSV (Comma Separated Values):** The high-speed standard for massive, flat datasets (e.g., importing 100,000 historical inventory movements). It strips out formatting overhead, allowing the backend stream-parser to process gigabytes of data with minimal memory footprint.
*   **JSON (JavaScript Object Notation):** The format for developers and system integrations. Ideal for deeply nested, hierarchical data (e.g., a massive multi-level Bill of Materials) where flat CSVs fail to capture parent-child relationships effectively.
*   **PDF (Portable Document Format):** Strictly an export-only format. Used when data must be frozen in an immutable, paginated, and highly formatted state for external compliance audits, board meetings, or official dispatch.

## 3. The Import Engine & Workflow

Mass data ingestion is one of the highest-risk operations in an ERP. The import engine utilizes a strict, multi-step pipeline to prevent database corruption.

### 3.1 The Import Wizard
Users are guided through a rigid, 5-step process:
1.  **Upload:** User selects the target entity (e.g., "Customers") and uploads the file.
2.  **Mapping:** The system attempts to auto-align the file's columns to the database schema. The user manually corrects any mismatched columns.
3.  **Dry Run (Pre-Flight Validation):** The system parses the entire file against the database *in memory*, without committing any changes.
4.  **Execution:** If validation passes, the system commits the data. For large files ( $> 1,000$ rows), this is automatically pushed to a background worker, allowing the user to navigate away.
5.  **Results Summary:** Displays total rows processed, total succeeded, and total failed.

### 3.2 System Templates
*   To bypass the mapping step entirely, users can download a pre-formatted **Import Template** for any entity.
*   These Excel templates are dynamically generated by the ERP, complete with mandatory column headers highlighted in red, and Data Validation dropdowns populated with live ERP data (e.g., a "Currency" column will only allow the user to select active currencies from the system).

### 3.3 Strict Validation
The Dry Run phase executes three tiers of validation:
1.  **Type Validation:** Does the "Weight" column contain letters instead of numbers?
2.  **Referential Integrity:** Does the "Supplier_ID" in the spreadsheet actually exist in the ERP's database?
3.  **Business Rules:** Is the imported "Sales Order Date" in a financial period that has already been closed by Accounting?

### 3.4 Error Reports & Resolution
*   If the validation phase detects errors, the system blocks the import and generates a downloadable **Error Report**.
*   This report is an exact copy of the user's uploaded Excel/CSV file, but it appends a new column at the end: `[ERP_ERROR_REASON]`.
*   The user simply opens the error report, reads the specific reason on row 45 (e.g., "Invalid Tax Code"), fixes the typo, and re-uploads that exact error file.

### 3.5 Rollback Strategy (Atomic Transactions)
*   **All-or-Nothing (Atomic):** For critical financial data (e.g., Journal Entries), the import is wrapped in an atomic database transaction. If row 9,999 out of 10,000 fails, the entire transaction is rolled back. Zero rows are saved.
*   **Partial Success (Configurable):** For non-critical master data (e.g., importing 5,000 new CRM Leads), the user can toggle "Skip Errors". The system will successfully save the 4,900 valid rows and reject the 100 invalid rows, generating an error report exclusively for the failures.

## 4. The Export Engine

### 4.1 Export Templates & Custom Views
*   Users rarely want a raw dump of every database column. The module allows users to define **Export Templates** (e.g., "Monthly Sales Summary").
*   Users select specific columns, define the order, apply filters (e.g., `Date > Last Month`), and save the layout. Future exports require a single click.

### 4.2 Background Generation
*   When a user requests a massive export (e.g., "All Inventory Movements for 2025" - potentially millions of rows), the system prevents browser timeouts by immediately returning an "Export Queued" notification.
*   The file is compiled by a background worker. Once ready, the user receives an in-app notification with a secure, expiring download link.

## 5. Security & Permissions

Mass export is the primary vector for corporate data theft. The module enforces a draconian security perimeter.

### 5.1 Role-Based Access Control (RBAC)
*   **Import Permissions:** Handled per module (e.g., `IMPORT_CUSTOMERS`, `IMPORT_INVENTORY`). Usually restricted to IT, Data Migration Specialists, and Department Heads.
*   **Export Permissions:** Highly decoupled from standard "View" permissions. A user may have `VIEW_CUSTOMERS` to do their daily job, but they are explicitly denied `EXPORT_CUSTOMERS` to prevent them from downloading the entire client list to a USB drive before leaving for a competitor.

### 5.2 Audit Logging
*   Every single import and export action is permanently written to the Global Audit Log.
*   **Export Audits:** The system records *Who* exported the data, *When*, the *IP Address*, the exact *SQL Query/Filters* used, and the total *Row Count* extracted. If a massive data exfiltration event occurs, Security teams can instantly trace the footprint.
