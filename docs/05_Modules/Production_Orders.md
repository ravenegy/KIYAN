# Module Specification Document
## Production Order Management Module (MES Core)
**System:** KIYAN Factory ERP (Adaptable for Generic Enterprise Manufacturing)
**Prepared by:** Enterprise ERP Solution Architect
**Date:** July 2026
**Architecture Environment:** Modular Monolith, Offline-First, SCADA/MES Integration Ready

---

## 1. Objective & Scope
This document outlines the official specification for the Production Order Management Module. Functioning as the Manufacturing Execution System (MES) core of the ERP, this module tracks the physical transformation of raw materials into finished goods on the shop floor in real-time. It governs the entire execution lifecycle: from machine setup and material consumption to yield recording and waste tracking.

The architecture is explicitly designed with an API-first and event-driven foundation to seamlessly support future integrations with hardware-level MES, PLCs (Programmable Logic Controllers), and SCADA (Supervisory Control and Data Acquisition) systems.

## 2. Feature Specifications

### 2.1 Production Orders & Status Management
*   **Purpose:** The central execution document directing the shop floor to produce a specific quantity of a product using a specific recipe.
*   **Workflow:** Generated by Planning Module ➔ Status: `PLANNED` ➔ Manager Releases ➔ Status: `RELEASED` ➔ Operator Starts ➔ Status: `IN_PROGRESS` ➔ Operator Finishes ➔ Status: `COMPLETED` ➔ Finance Reviews ➔ Status: `CLOSED`.
*   **Business Rules:** A Production Order MUST be linked to exactly one `ACTIVE` Recipe Version. Financial costs cannot be settled until the order is `CLOSED`.
*   **Validation:** Cannot transition to `RELEASED` if required materials are missing (unless an executive override is applied).
*   **Dependencies:** Production Planning, Recipes (BOM), Inventory.
*   **Notifications:** Alert Warehouse to pick and stage materials when an order is `RELEASED`.
*   **Reports:** Daily Production Schedule, Order Status Matrix.
*   **KPIs:** Schedule Adherence (Planned vs. Actual start times).
*   **Audit Logs:** Every status transition is immutably logged with the triggering User UUID and timestamp.

### 2.2 Machine & Operator Assignment
*   **Purpose:** To strictly allocate and record which physical and human resources are executing the order.
*   **Workflow:** Production Manager assigns Work Centers (Machines) ➔ Operator logs into the Work Center terminal ➔ System binds the Operator and Machine to the active Production Order.
*   **Business Rules:** An operator cannot be assigned to a machine they are not certified to run. A machine cannot be assigned if it has an active `MAINTENANCE` status.
*   **Validation:** Enforces 1:1 or M:1 Operator-to-Machine mappings based on Work Center configuration.
*   **Dependencies:** HR (Certifications/Attendance), Maintenance (Machine Status).
*   **Notifications:** Alert Maintenance if an operator reports a machine fault during assignment.
*   **Reports:** Operator Utilization Rate, Machine Allocation Log.
*   **KPIs:** Labor Efficiency Variance (Standard vs. Actual labor hours).
*   **Audit Logs:** Records exact login/logout times of operators against specific machines.

### 2.3 Execution Control (Start, Pause, Resume, Stop, Complete)
*   **Purpose:** To capture precise time-series data of manufacturing operations for OEE (Overall Equipment Effectiveness) and labor costing.
*   **Workflow:** Operator interacts with the shop-floor UI to trigger state changes.
    *   **Start:** Begins the runtime clock.
    *   **Pause:** Halts the clock. Requires a Reason Code (e.g., "Cleaning", "Material Shortage", "Break").
    *   **Resume:** Restarts the clock.
    *   **Stop/Complete:** Ends the operation and triggers final yield calculations.
*   **Business Rules:** Material cannot be systematically consumed unless the order is `IN_PROGRESS`.
*   **Validation:** Cannot `Resume` an order that is not `PAUSED`. Cannot `Complete` an order with zero yield unless fully rejected.
*   **Dependencies:** Core Application state engine.
*   **Notifications:** Alert Production Manager if an order is `PAUSED` for more than 15 minutes.
*   **Reports:** Machine Downtime Analysis (by Reason Code).
*   **KPIs:** OEE (Availability factor), Mean Time to Repair (MTTR) if paused for faults.
*   **Audit Logs:** Every micro-state transition is logged to calculate exact runtime vs. idle time.
*   **Future MES Integration:** These state changes will be triggerable directly via MQTT payloads from PLC relays (e.g., when the roaster motor turns on, the system automatically transitions to `IN_PROGRESS`).

### 2.4 Material Consumption
*   **Purpose:** The physical and logical deduction of raw/packaging inventory during manufacturing.
*   **Workflow:** Operator scans Staging Bin ➔ Scans RM Lot Barcode ➔ Scans into Machine Hopper ➔ System deducts Inventory and attaches the Lot to the Production Order's traceability record.
*   **Business Rules:** Strictly enforces FEFO/FIFO routing. Backflushing (auto-consuming materials at the end based on yield) is supported for continuous-flow processes, but explicit barcode scanning is mandated for critical/traceable ingredients.
*   **Validation:** System hard-blocks the consumption of `QUARANTINED` or `EXPIRED` lots, and lots not present on the active BOM.
*   **Dependencies:** WMS, Inventory, Recipes (BOM).
*   **Notifications:** Real-time alert if a scanned Lot is nearing its expiry date.
*   **Reports:** Material Variance Report (Expected BOM Consumption vs. Actual Scanned Consumption).
*   **KPIs:** Material Yield Variance.
*   **Audit Logs:** Exact timestamp and User ID of the operator who scanned the Lot.

### 2.5 Waste Recording (Scrap)
*   **Purpose:** Tracking non-usable material loss during the production process for financial write-offs and process improvement.
*   **Workflow:** Operator weighs waste (e.g., burnt sesame, dropped jars) ➔ Enters quantity into the system ➔ Selects Scrap Reason Code ➔ Submits.
*   **Business Rules:** Scrap quantities are added to the total material consumed but do not contribute to the Finished Good yield. Financial value is routed to a COGS/Scrap expense account.
*   **Validation:** Scrap quantity cannot exceed total material consumed.
*   **Dependencies:** Accounting, Inventory.
*   **Notifications:** Immediate alert to Factory Manager if scrap for a single order exceeds the BOM's defined `Waste Percentage` tolerance.
*   **Reports:** Scrap Pareto Chart (by Machine, Reason Code, Operator).
*   **KPIs:** Scrap Rate (%), First Pass Yield (FPY).
*   **Audit Logs:** Tracks the operator who logged the waste and the justification code.

### 3.6 Yield Calculation & Finished Goods (FG)
*   **Purpose:** Capturing the usable output of the manufacturing process and injecting it into active inventory.
*   **Workflow:** Production completes ➔ Operator/Scale transmits final net weight/count ➔ System generates a new unique Finished Good `Lot ID` ➔ System prints FG Barcode/QR labels ➔ Operators palletize and transfer to FG Warehouse.
*   **Business Rules:** The newly generated FG Lot permanently inherits the lineage (child-to-parent relationship) of every Raw Material Lot consumed during the order.
*   **Validation:** Output quantity must be within an acceptable statistical tolerance of the (Inputs - Waste).
*   **Dependencies:** Inventory (Lot Generation), WMS (Put-away tasks), Quality.
*   **Notifications:** Alert QC that a new FG Lot has been produced and requires release testing (defaults to `QUARANTINED` for food manufacturing).
*   **Reports:** Electronic Batch Record (EBR), Production Traceability Report (Forward/Backward).
*   **KPIs:** Manufacturing Yield (%), OEE (Performance factor).
*   **Audit Logs:** Immutable record linking the FG Lot to the specific Production Order and timestamp.

### 3.7 Reject & Rework
*   **Purpose:** Managing output that fails in-line quality checks but is salvageable.
*   **Workflow:** Operator identifies defect ➔ Logs as `REJECT` (diverting from FG Yield) ➔ System generates a logical `Rework Order` linked to the original Production Order.
*   **Business Rules:** Rework Orders must utilize specific Rework Recipes/Routings (e.g., re-milling coarse paste). Costs incurred during rework are accumulated against the original order to calculate the true cost of poor quality (COPQ).
*   **Validation:** Rejected material must be physically moved to a designated Rework Zone before the Rework Order can be started.
*   **Dependencies:** Quality, Recipes (Rework BOMs).
*   **Notifications:** Alert QC and Production Manager of a significant Reject event.
*   **Reports:** Cost of Poor Quality (COPQ) Analysis, Rework Frequency by Product.
*   **KPIs:** Rework Rate (%), OEE (Quality factor).
*   **Audit Logs:** Full lifecycle tracking of the rejected lot through its rework transformation.

## 3. Future MES & IIoT Integration

To ensure the ERP remains future-proof, the Production Order Management Module is designed to act as an integration hub for Industrial Internet of Things (IIoT) devices:

*   **API-First Endpoints:** Every action (Start, Stop, Consume, Yield) is exposed via secure internal REST/gRPC APIs.
*   **MQTT Broker Support:** The architecture supports listening to MQTT topics published by machine PLCs.
    *   *Example:* A digital scale at the end of the Tahini line publishes a weight payload via MQTT. The ERP automatically catches this payload, calculates the Yield, generates the FG Lot, and prints the ZPL barcode label, entirely eliminating manual operator data entry.
*   **Headless Execution:** The shop-floor UI is strictly a presentation layer. Complex factories can replace the KIYAN UI with their own SCADA screens while still utilizing the ERP's underlying MES logic.
